# Part 4 – LLM-Based Batch Scoring System

## Overview

In this part, I developed a simple LLM-powered application that analyses housing records using Google's Gemini API. The program reads records from the cleaned dataset, sends them to the language model, validates the responses, and saves the results in JSON format.

The objective was to demonstrate how a Large Language Model can be integrated into a structured data processing workflow while maintaining reliable and consistent outputs.

---

## Dataset

The input dataset used in this part is the `cleaned_data.csv` file generated during Part 1 of the project.

Only the first three records are processed for demonstration purposes.

---

## Project Workflow

The application follows the steps below:

1. Load the cleaned housing dataset.
2. Read the first three records.
3. Generate a prompt for each record.
4. Send the prompt to the Gemini model.
5. Receive the prediction in JSON format.
6. Validate the returned JSON using a predefined schema.
7. Check for basic Personally Identifiable Information (PII).
8. Save the validated results to a JSON file.

---

## Prompt Design

A separate system prompt is used to define the behaviour of the model.

Each housing record is inserted into a prompt template before being sent to Gemini.

The model is instructed to return only structured JSON without any additional explanations or markdown formatting.

---

## JSON Validation

The response generated by the model is validated using the `jsonschema` library.

The expected JSON contains:

- Predicted Price Level
- Overall Condition
- Reason

Any response that does not match the expected structure is rejected.

---

## Guardrails

A simple guardrail was added before sending data to the model.

The program checks for:

- Email addresses
- Phone numbers
- 12-digit identification numbers

If any of these patterns are detected, the corresponding record is skipped.

---

## Temperature Comparison

The model was tested with two temperature values.

- **Temperature = 0** produced consistent and repeatable responses.
- **Temperature = 0.7** generated slightly different explanations while keeping the prediction unchanged.

Since this project requires stable and predictable outputs, the final implementation uses a temperature of **0**.

---

## Output

The validated predictions are stored in:

```
outputs/results.json
```

Each record contains:

- Record Number
- Predicted Price Level
- Overall Condition
- Reason


---

## Running the Project

1. Create a `.env` file and add your Gemini API key.

```
GEMINI_API_KEY=YOUR_API_KEY
```

2. Make sure `cleaned_data.csv` is present in the project folder.

3. Run the application.

```bash
python main.py
```

4. To compare different temperature values, run:

```bash
python temperature_test.py
```

---

## Conclusion

This part demonstrates how an LLM can be integrated into a practical machine learning workflow. By combining prompt engineering, schema validation, simple guardrails, and structured JSON output, the application produces reliable responses that are suitable for downstream processing. The project also highlights the importance of validating model outputs before using them in real-world applications.
